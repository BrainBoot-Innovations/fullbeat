3. Login & First-Time Setup

Signing In

  1. Open fullbeat.brainboot.co.in in your browser.
  2. Enter the email and temporary password provided by your Admin.
  3. Click "Sign In".

First Login — Password Change

On your very first login, FullBeat will require you to set a new password before proceeding.

Password Requirements Minimum 8 characters, at least 1 uppercase letter (A–Z), and at least 1 number (0–9).
  1. Enter your New Password.
  2. Re-enter it in Confirm Password.
  3. Click "Set Password".
  4. You will be redirected to the Dashboard.

Navigation

After login, the top navigation bar shows all available modules. On mobile, tap the hamburger menu (☰) to reveal the navigation tabs.

The header bar shows: your name, your tester code (e.g. T01), the active project selector, and a Logout button.

4. Dashboard

The Dashboard is your command center. It shows the live status of the currently active test plan.

4.1 Active Plan Card

Displays the plan name, type, environment, and total test cases. If the plan was activated with execution details, you will also see:

  • Run Purpose — Why this test run is happening
  • Deadline — When execution must be completed (turns amber within 24 hours, red + OVERDUE if passed)
  • Required Team — Who is expected to participate
  • Special Instructions — Any additional guidance from the admin

4.2 Verdict Banner

A large color-coded banner that instantly tells you the release readiness:

VerdictColorMeaning
NOT STARTED GrayNo test cases have been executed yet.
IN PROGRESS AmberExecution is underway (X of Y completed).
GO FOR PRODUCTION GreenAll tests executed, zero failures. Safe to release.
CONDITIONAL GO YellowAll tests executed, only negative-test failures (5 or fewer). Release with caution.
NO-GO RedFunctional failures exist, or more than 5 negative failures. Do not release.

5. Test Case Repository

The Repository is the master list of all test cases for your project. Every test plan draws from this pool.

5.1 Browsing & Filtering

Use the filter bar at the top to narrow down test cases:

FilterOptions
ModuleDynamic — populated from your test cases (e.g. Authentication, Dashboard, Settings)
CategoryAll, Smoke, Sanity, Regression, Functional, Negative, Performance
SearchFree text — searches scenario descriptions
Previously FailedCheckbox — shows only test cases that failed in a prior run

5.2 Adding a Test Case

  1. Click "+ Add Test Case".
  2. Fill in the required fields:
    • Module — The feature area (e.g. "Authentication")
    • Category — smoke / sanity / regression / functional / negative / performance
    • Scenario — Short description of what is being tested
    • Steps — Step-by-step instructions to execute
    • Expected Result — What should happen if the test passes
  3. Optionally fill in Preconditions and Test Data.
  4. Click "Save".

5.3 Bulk Upload (JSON)

For large batches of test cases, use the JSON upload feature:

  1. Click "Upload JSON".
  2. Step 1 — Select File: Drag and drop a .json file or click to browse. Choose the import mode:
    • Fresh Import — All entries are added as new test cases
    • Update Existing — Matches by TC Index; updates if found, inserts if new
  3. Step 2 — Preview: Review the list. Remove any unwanted entries.
  4. Step 3 — Result: See success/failure count. Optionally click "Map to Plan" to immediately create a test plan from the imported cases.
JSON Format Each test case in the JSON array should have: module, category, scenario, steps, expected_result. Optional: tc_index, preconditions, test_data.

Step 2 — Auto-Select

FullBeat automatically picks test cases based on the plan type:

  • Smoke → Only cases with category = "smoke"
  • Sanity → Cases with category = "smoke" or "sanity"
  • Regression → All active test cases
  • Custom → Starts empty

If you recently bulk-uploaded test cases, they will be pre-selected here automatically.

Step 3 — Customize

Add or remove individual test cases. The counter at the bottom shows how many are selected.

Step 4 — Assign Testers

For each test case, pick a tester from the dropdown. Or click "Auto-Assign (Round Robin)" to distribute cases evenly across all active testers.

Step 5 — Review

Verify the plan summary: ID, name, type, environment, version, test case count, and tester assignments. Click "Create Plan" to save as a Draft.

7. Plan Activation — Run Setup

When an Admin or Lead clicks "Activate" on a draft plan, a modal appears requiring key execution details before the plan goes live.

Why This Step Matters Activation details ensure every tester knows: what this run is for, who is expected to participate, and when it must be completed. This information flows through to the Dashboard and the Failure Analysis Report.

Activation Modal Fields

FieldRequiredDescription
Run PurposeYesWhy this test run is happening. Examples:
• "Sprint 14 release validation"
• "Hotfix verification for login bug"
• "Pre-production smoke test before v2.1 go-live"
Required Team MembersYes (at least 1)Checklist of all active testers. Those already assigned in the plan are pre-checked and tagged with Assigned in plan. Select additional members if needed.
Deadline to CompleteYesDate & time by which all execution must finish. Defaults to 3 days from now. Must be in the future.
Special InstructionsNoAny extra guidance. Examples:
• "Focus on payment module first"
• "Use staging environment only"
• "Retest previously failed cases from Sprint 13"

After Activation

  • The plan status changes from Draft to Active.
  • The plan appears in the Execution page dropdown — testers can now start executing.
  • The Dashboard shows the run purpose, deadline (with color-coded urgency), and required team in the Active Plan card.
  • The Plan Detail modal shows all execution details in a green info box.

8. Test Execution

This is where testers do the actual work — running test cases and recording results.

8.1 Getting Started

  1. Go to the Execution page.
  2. Select an Active or Completed plan from the dropdown.
  3. The test cases for that plan load in the table below.

8.2 Filters

FilterOptions
ModuleDynamic — based on test cases in the selected plan
StatusAll, Pending, Pass, Fail, Blocked, Skipped
My Assignments OnlyCheckbox — shows only test cases assigned to you

8.3 Executing a Test Case

  1. Click on a scenario name to expand and see the full steps and expected result.
  2. Perform the test as described in the steps.
  3. Click one of the status buttons:
    • PASS — Test passed as expected
    • FAIL — Test did not produce the expected result (auto-opens bug modal)
    • BLOCKED — Cannot execute due to a dependency or environment issue
    • SKIP — Intentionally not testing this cycle
  4. Optionally click "Add" in the Remarks column to leave notes.

8.4 Filing a Bug (on FAIL)

When you click FAIL, a bug creation modal automatically opens:

FieldAuto-filled?Description
Bug CodeYesAuto-generated as T01-001 (your tester code + sequence number)
TitleYesPre-filled as "[TC-XXX] Scenario name" — edit if needed
SeverityNoSelect: Critical / Major / Minor / Cosmetic
DescriptionNoDescribe what went wrong, steps to reproduce, actual vs expected
Screenshot URLNoPaste a link to the screenshot (hosted on GitHub, S3, Imgur, etc.)

Click "File Bug" to save. The bug is linked to the test case and appears on the Dashboard and Bugs page.

Screenshot Tip Screenshots are shown inline in the Failure Analysis Report on the Dashboard. Use a reliable image host so the team can view them directly without opening separate links.

8.5 Progress Tracking

The progress bar at the top of the execution page updates in real-time as you mark test cases. It shows: "X / Y executed" with a percentage bar.

9.3 Bug Detail Modal

Click "View" on any bug to open the detail modal. It has three sections:

Section 1 — Bug Information (Read-Only)

  • Bug code, title, severity, module, related test case
  • Full description
  • Screenshot (if provided)
  • Filed by and date

Section 2 — Fix Details (Lead/Admin)

  • Fix Status — Update to: In Progress, Fixed, Won't Fix
  • Fixed By — Name of the developer who fixed it
  • Fix Description — What was changed to resolve the bug

Section 3 — Retest (Tester/Lead)

  • Retest Status — Pending, Pass, or Fail
  • If retest = Pass, the fix status automatically advances to Closed
  • If retest = Fail, the bug remains open for another fix cycle

Timeline

The bottom of the modal shows a visual timeline of all status transitions with timestamps.

10. Failure Analysis Report

This section appears automatically on the Dashboard when there are failures or open bugs. It is designed to help you read and understand all failures at a glance without opening individual bugs.

When Does It Appear? The Failure Analysis Report card is hidden when there are zero failures and zero open bugs. As soon as any test case fails or a bug is filed, it becomes visible.

10.1 What It Shows

Execution Context Banner

If the plan was activated with execution details, you see a blue info box at the top showing: Run Purpose, Deadline, Required Team, and Special Instructions.

Failures Grouped by Module

Each module with failures gets its own collapsible section:

  • Module header — Module name + red badge showing failure count
  • Failed test cases — For each failure:
    • TC index and scenario name (in red)
    • Tester who executed it
    • Execution remarks (if any)
  • Related bugs — For each bug linked to that test case:
    • Bug code and title
    • Severity badge (color-coded: Critical=red, Major=orange, Minor=yellow, Cosmetic=gray)
    • Current fix status
    • Full description
    • Reporter name and date
    • Screenshot rendered inline — The actual image loads directly in the report. Click it to open full-size in a new tab.
  • If a failure has no bug filed, it shows: "No bug filed for this failure."

Orphan Bugs

Bugs that exist for modules with no failed test cases (e.g. filed outside the execution flow) are also shown at the bottom.

10.2 How to Use the Report

  1. Scan the badge at the top right — it tells you total failures and total bugs.
  2. Read module by module — see which areas of the application have the most issues.
  3. Check screenshots — view the actual failure evidence inline without opening each bug.
  4. Review bug status — see which bugs are still open vs. being fixed.
  5. Share the dashboard — the report is a live, always-current view that stakeholders can check anytime.
Pro Tip: Attach Screenshots to Every Bug The Failure Analysis Report is most useful when every bug has a screenshot URL. When filing a bug on the Execution page, always paste a screenshot link. The image will render directly in the report, making it easy for leads and admins to scan all failures visually.

11. Admin — User Management

The Admin page is only visible to users with the Admin role.

11.1 Creating a New User

  1. Click "+ Create User".
  2. Fill in the fields:
    • Email — The user's login email
    • Display Name — Their name as shown in the app
    • Tester Code — Unique code (format: T01, T02, etc.)
    • Role — Engineer, Lead, or Admin
    • Temporary Password — Must meet password requirements (8+ chars, 1 uppercase, 1 number)
  3. Click "Create User".
  4. Share the email and temporary password with the user. They will be forced to change the password on first login.

11.2 Managing Users

The user table shows all accounts with their name, tester code, role, status (Active/Inactive), and password status (Pending/Done).

  • Deactivate — Prevents the user from logging in. They are not deleted, just disabled.
  • Activate — Re-enables a previously deactivated user.

14. Tips & FAQ

Common Questions

Q: I can't see the Admin tab.

The Admin tab only appears for users with the Admin role. If you need admin access, ask your team admin to update your role.

Q: How do I re-execute a test case that I already marked?

Simply click a different status button on the same row. The status will update and the result is overwritten. The test case's Last 3 Runs history keeps track of previous results.

Q: Can I file a bug without marking a test case as FAIL?

No. Bugs are created through the execution flow when you click FAIL. This ensures every bug is linked to a specific test case for traceability.

Q: Where do I host screenshots for bugs?

FullBeat accepts any HTTP/HTTPS URL for screenshots. Common options: GitHub repository images, Imgur, Google Drive (with public sharing), or any cloud storage with public URLs.

Q: What does "Previously Failed" mean in the repository?

If a test case failed in any of its last 3 executions across any plan, it is flagged as "Previously Failed". This helps identify flaky or problematic areas.

Q: Can I edit a plan after it's activated?

No. Once activated, the plan is locked. If you need changes, complete or archive the current plan and create a new one.

Q: What happens when the deadline passes?

The deadline is shown in red with "OVERDUE" on the Dashboard. Execution can still continue — the deadline is informational, not enforced.

Keyboard & Browser Tips

  • FullBeat works best in Chrome, Edge, or Firefox (latest versions).
  • Use Ctrl + F in the browser to search on any page.
  • Exported CSV files can be opened directly in Excel or Google Sheets.
  • The JSON export is useful for custom analysis in Python, Node.js, or any data tool.
